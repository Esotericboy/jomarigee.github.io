    Information Gathering

    The target IP address is provided when the machine is deployed.

    Target: 10.10.158.221

    Scanning/Enumeration

    Running an nmap scan on the target I get back a lot of ports, the first being 21 allowing me to login anonymously into the Windows FTP file system. Then I see there is website out there along with ports 135,139, and 445 all pertaining to Windows. Lastly, I see that Windows Server 2008 R2 -2012 is in use. 

    ➜ nmap -A 10.10.10.152 -o nmap.txt

    Then running a vulnerability script scan it took a little time, but it gave SO MUCH output that I can't include it all on this page. Most of the output came from viewing the source code of the index page. Along with the potential for CSRF vulns and a potential CVE for Source Code Disclosure (CVE-2010-2333). It's also worth noting that the file extension HTML is used instead of html, which isn't important I think yet. 

    ➜ gobuster dir -u http://10.10.10.152 -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt -n -e -x php,html,txt -t 30

    Trying to run a Gobuster scan I get told that the box will only show status codes for 302, which is used for redirections. This can be validated because if I try to go from /index.html to index.html I'll get redirected to /public/login.htm?loginurl=%2Findex.html&errormsg= . I figured out thanks to this great article how to route my traffic through Burp Suite for accurate results, but honestly this was a rabbit hole and waste of time.

    User Flag

    Logging into the FTP server as the user Anonymous I see that the filesystem is set up just how it would be in Windows, makes sense because this is a Windows box.

    Going into the directory /Users/Public I see a file called user.txt which could quite possibly be the user flag! However, I can't download it from the FTP server no matter what command I use.

    Looking through my notes on FTP I see that sometimes you can browse to the FTP servers GUI. I navigate to the FTP server with the command below, go to the location of the user.txt file. Low and behold Im able to see the flag right there in the browser! 

    ➜ ftp://10.10.10.152/Users/Public/

    Later on I went back and tried to download the user flag to my machine and it worked unlike last time, strange.

    Gaining Credentials

    I can't login to the main website because I don't have any credentials to leverage, so I turn my head back to FTP where I find some nice looking files in C:\User\All Users\Paessler\PRTG Network Monitor. These caught my eye because they potentially could hold the login credentials that I need for the PRTG login portal.

    Since this was an off the shelf product I did some research to see what the default login credentials would be and found prtgadmin to be the username. Armed with this information I grepped the three files I had downloaded for any mention of that username for hope that it would be followed by a password.

    ➜ cat * | grep -A 20 -B 20 "prtgadmin"

    To which I was met with a oh so sweet password, that allowed me to login to the application now. Actually had to change the password to PrTg@dmin2019 to login since the box came out in 2019.

    Root Flag

    After browsing around the website for a little bit I took to Searchsploit, seeing if I could find a potential exploit for this service even though I didn't know the service version. To my surprise I found an RCE exploit that did require authentication, which isn't a problem since I'm already in.

    Running the exploit it tells me what I need to do to actually get the RCE, which is nice. All I need to do is capture an authenticated request with Burp then pass that to the script and boom a shell lol.

    I kept trying every which way to run the exploit, but nothing seemed to be working. Thanks to this writeup it made me realize that the exploit wasn't fully copied to searchsploit correctly, a problem I'd never encountered before. So I downloaded it straight from ExploitDB and ran the command again.

    Now with the exploit coming straight from DB it was able work successfully and gave me my new username and password and I can use with psexec to login and finish this box up.

    Using psexec along with good ol python3 I get into the system with the privileges of authority! 

    ➜ python3 /usr/share/doc/python3-impacket/examples/psexec.py pentest:"P3nT3st!"@10.10.10.15

    Now that I'm logged in the system was being weird and not letting me enter normal commands, so I had to navigate the slow way, but eventually got to the root flag!

    Finished

    I loved this machine even though it was a fairly straightforward box. I enjoyed learning how to route Gobuster traffic through Burp even though it was a rabbit hole. Overall a fun machine to do. ☺

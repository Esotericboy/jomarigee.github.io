    Machine Information

    DC-9 is the ninth and final machine in the DC series. Like the previous boxes before it the one and only goal is to get root access and read the root flag.

    You can find this Machine Here

    Information Gathering

    The target IP address isn't provided upon machine start, so I used the tool netdiscover to find the IP of the machine.

    ➜ sudo netdiscover -r 192.168.0.1/24

    Scanning/Enumeration

    Running an nmap scan on the IP address I see that only port 80 is open with an example page on it. Interesting that there aren't any other ports open, yet.

    ➜ nmap -A 10.10.0.0

    Looking at the scan we can see the following ports and services are open:

    • ● Port 80/tcp - HTTP - (httpd 2.4.38)

    Checking out the results of the gobuster scan I get back a lot of directories which is nice, but first I need to check out the main website then work back from there.

    ➜ gobuster dir -u http://192.168.0.68 -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt -n -e -x php,html,txt -t 30

    Crawling Main Website

    Looking at the main domain its a pretty basic Drupal like almost site with just three different tabs on it.

    When visiting the Display all information section I see a long list of names, ID's along with much more. This all could be useful for trying to brute force a login page.

    Going to the search tab, its exactly what you would think it is. A tab to search for information and pulls searches for different employees.

    The manage page is a login portal which is perfect for the amount of potential credentials that were in the Display All information section.

    Weird Functionality

    When navigating to /logout.php I'm told that I'm already admin. Along with the message "File Does Note Exist" , which is weird to say the least. Now I can't go to /manage.php without being told I'm the admin

    SQL Injection

    I tried to brute force the login portal, but it didn't work at all, so the only thing possible now is an SQL Injection in the Search Bar. I used Burp Suite to capture the test request and save it in a text format to use SQLmap with.

    Passing the POST request to SQLmap I find out the target is vulnerable to an SQLi, which is awesome! Along with seeing that it's vulnerable, I also see the service version of SQL being used

    ➜ sqlmap -r DC.txt --dbs --batch

    Next I dump the different databases and see lots of information about the different staff that work there, but most importantly I get a username of admin along with the hashed password.

    ➜ sqlmap -r DC.txt -D Staff --dump-all --batch

    Then I take that hashed password to CrackStation and let it go to work so I can get the password from the intentionally weak MD5 hash.

    Port Knocking

    However, even though getting this password was cool, it doesn't help accomplish anything because I'm already admin. So I look back at the error message and realize that this could be a LFI vulnerability because it's looking for a configuration file.

    ➜ http://192.168.0.68/manage.php?file=../../../../etc/passwd

    Reading over the large output that was given I notice that it looks like a Linux file system which would make sense because on my nmap scan I didn't see port 22 open, which is weird for a web server, time to knock ssh down

    ➜ http://192.168.0.68/manage.php?file=../../../etc/knockd.conf

    Next I knocked down the port via knock and then used nmap to verify that it was open.

    ➜  knock 192.168.0.68 7469 8475 9842

    Brute-Forcing SSH

    Going back to the SQLMap output that I got I made two list, one from the passwords and one from the usernames.

    Then I used Hydra to Brute Force SSH to get the login credentials.

    ➜ hydra -L usernames.txt -P passwords.txt 192.168.0.68 -t 4 ssh

    Logged in as the user janitor right off the bat I see this weird text file with a passwords file. Which was found in a "hidden" directory. One of the passwords here is for the user Phoebe, so I figure the other passwords have to be for other users. This will be perfect for brute forcing SSH via Hydra.

    I brute forced SSH with the passwords list and was able to login as the user Fred!

    Privilege Escalation

    I enter sudo -l and to my surprise see that I don't need a password to become root on this box!

    I find a python file that might be the exploit I need to get root access.

    Looking over this file more in depth it is confusing to me because I don't understand python yet.

    So I decide it would be the easiest to just make a new user with root privileges and append them to /etc/passwd file.

    Then once I have made the new user I just login as them and now I'm root.

    Root Flag

    Now that I am root, it's just a simple navigation to the root directory and catting out the root flag file to finish off the box.

    Finished

    I loved this box because it was my first VulnHub machine and I've never knocked down a port so that was fun. This box was great and the creator of this series is amazing. ☺
